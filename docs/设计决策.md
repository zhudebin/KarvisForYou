# KarvisForAll — 设计决策记录

## DD-001: UserContext 全链路贯穿

**背景**：Karvis 是单用户架构，state/memory/文件路径都是全局常量。改造为多用户后，需要一种方式让每条消息的处理链路"知道当前是谁"。

| 方案 | 优点 | 缺点 |
|------|------|------|
| 全局变量 + thread-local | 改动少 | 线程切换或异步场景易串用户 |
| 函数参数传递 ctx 对象 | 显式、可追踪、无并发隐患 | 改动面大——每个函数签名都要加 ctx |

**决策**：方案二。定义 `UserContext` 类，封装用户的全部路径属性（`base_dir` / `quick_notes_file` / `state_file` / `memory_file` 等 17 个路径），从入口 `app.py` 创建，通过函数参数一路传递到 `brain.process()` → Skill handler → Storage IO。

```
企微消息 → app.py 解密得到 user_id
  → get_or_create_user(user_id) 得到 UserContext
  → brain.process(payload, ctx, send_fn)
    → read_state_cached(ctx)
    → load_memory(ctx)
    → handler(params, state, ctx)
  → send_wework_message(user_id, reply)
```

**理由**：显式参数传递是最安全的多用户隔离方式。ctx 对象不可变（路径在构造时确定），不存在并发读写问题。虽然改动面大（每个 skill handler 签名都加 ctx），但一次性改完后，后续新增 skill 只需遵循约定即可。

## DD-002: 本地存储 vs OneDrive

**背景**：Karvis 使用 OneDrive Graph API 存储所有数据（state/memory/笔记），用户可通过 Obsidian + remotely-save 在手机端查看。KarvisForAll 需要为多个用户存储数据。

| 方案 | 优点 | 缺点 |
|------|------|------|
| 沿用 OneDrive（管理员账号） | 与 Karvis 架构一致 | 多用户数据存一个人的 OneDrive 有隐私问题；网络延迟不可控 |
| 每用户配自己的 OneDrive | 数据完全归属各用户 | 配置门槛极高，每人需要 OAuth 授权 |
| 本地磁盘 + Web 查看页面 | 零网络延迟；Web 页面替代 Obsidian 查看 | 不支持 Obsidian 浏览 |

**决策**：方案三。`storage.py` 固定使用 `LocalFileIO`，所有数据存储在本地 `data/users/{user_id}/` 目录。通过 Web 查看页面（8 个页面）替代 Obsidian 浏览。Docker volume 持久化数据目录。

```python
# storage.py — 一行切换
from local_io import LocalFileIO
IO = LocalFileIO
```

**理由**：
1. 多用户数据存管理员个人 OneDrive 有严重隐私问题
2. 本地读写 0ms vs OneDrive 2-33s，性能提升数量级
3. Web 查看页面提供更好的移动端体验（Tailwind CSS 响应式）
4. 保留 `IO` 别名和 `get_token()` 接口兼容，历史代码无需大改

## DD-003: Skill Handler 统一签名 `(params, state, ctx)`

**背景**：Karvis 的 skill handler 签名为 `(params, state)`。多用户改造后，handler 需要知道当前用户的文件路径。

| 方案 | 优点 | 缺点 |
|------|------|------|
| 通过 state 携带用户路径信息 | handler 签名不变 | state 职责不单一，污染数据结构 |
| 新增 ctx 参数 `(params, state, ctx)` | 语义清晰，ctx 专门负责路径 | 所有 handler 签名改动 |

**决策**：方案二。所有 skill handler 统一签名 `handler(params: dict, state: dict, ctx: UserContext) -> dict`。`skill_loader.py` 扫描 `skills/` 目录时验证签名。`brain.py` 调用时统一传入三参数。

**理由**：一次性改动，但语义清晰——`params` 是 LLM 决策参数，`state` 是用户运行时状态，`ctx` 是用户路径上下文。三者职责分明。内置的 `ignore` handler 也遵循此签名：`lambda params, state, ctx: {"success": True}`。

## DD-004: Per-user 缓存分区

**背景**：Karvis 的 state 缓存是全局单例 `_state_cache = {"data": ..., "expire_time": ...}`。多用户场景下必须按用户分区，否则用户 A 的缓存会被用户 B 覆盖。

| 缓存项 | Karvis（单用户） | KarvisForAll（多用户） |
|--------|------------------|----------------------|
| State 缓存 | `_state_cache = {data, expire}` | `_state_cache = {user_id: {data, expire}}` |
| Prompt 缓存 | `PromptCache` 全局单例 | 保持全局（Prompt 模板所有用户共享） |
| Memory 缓存 | 全局单份 | 按 user_id 分区 |

**决策**：State 和 Memory 缓存按 `user_id` 分区。Prompt 缓存保持全局共享（SOUL/SKILLS/RULES 是代码内常量，所有用户共用）。缓存键统一为 `_state_cache[user_id]`。

**理由**：Prompt 是系统级配置不含用户个人数据，共享节省内存。State/Memory 是用户私有数据，必须分区。TTL 策略沿用 Karvis（State 5 分钟，Prompt 30 分钟）。

## DD-005: Web Token 鉴权

**背景**：用户需要通过浏览器查看自己的数据。需要一种鉴权机制，要求：简单（2-3 人场景）、安全（不能看到别人数据）、低摩擦（不需要记密码）。

| 方案 | 优点 | 缺点 |
|------|------|------|
| 用户名+密码 | 传统方案 | 需要注册流程、记密码 |
| OAuth（企微扫码登录） | 安全 | 需要配置 OAuth 回调、实现复杂 |
| Token 令牌制 | 零密码、对话触发即用 | 令牌泄露有风险（但有过期机制） |

**决策**：Token 令牌制。用户在企微中说"给我查看链接"，Karvis 生成 UUID v4 令牌（默认 24h 有效），回复带 token 的 URL。

**令牌验证三级 fallback**：
1. `X-Token` HTTP Header
2. `karvis_token` Cookie
3. `token` URL Query Parameter

**令牌存储**：内存 + 持久化到 `_karvis_system/tokens.json`（进程重启后令牌依然有效）。`_tokens_lock` 保证并发安全。

**理由**：2-3 人小圈子场景，Token 令牌制是最低摩擦方案。对话式触发（"给我链接"）比登录页面更自然。24h 过期 + 不分享链接 = 足够安全。

## DD-006: 管理员后台鉴权

**背景**：管理员需要查看所有用户的使用情况、挂起/激活用户。需要独立于普通用户 Token 的鉴权机制。

**决策**：环境变量 `ADMIN_TOKEN` 配置固定管理密钥。管理员通过 `/web/admin` 页面输入此密钥登录。

**鉴权三级 fallback**：
1. `X-Admin-Token` HTTP Header
2. `admin_token` URL Query Parameter
3. `karvis_admin_token` Cookie

**理由**：2-3 人场景不需要复杂的 RBAC 权限体系。管理员只有部署者一人，一个固定密钥足够。通过 Cookie 保存登录状态，避免每次访问都要输入。

## DD-007: 成本控制 — 每日消息上限 + LLM 用量追踪

**背景**：管理员承担所有 LLM 费用。需要防止某个用户疯狂发消息导致费用失控。

**决策组合**：

1. **每日消息上限**：`DAILY_MESSAGE_LIMIT`（默认 50 条/人/天）。`increment_message_count()` 在跨天时自动重置。超限后返回"今天消息额度用完啦，明天再聊～"。
2. **LLM 用量追踪**：每次 `call_llm()` 记录 `{user_id, model, input_tokens, output_tokens, timestamp}` 到 `_karvis_system/usage_log.jsonl`。`_set_current_user()` 通过 `threading.local()` 暂存 user_id，供日志记录。
3. **不活跃用户跳过**：定时任务遍历前检查 `last_active`，超过 `INACTIVE_DAYS_THRESHOLD`（默认 7 天）不触发 LLM 类任务。

**理由**：三层防线。消息上限是硬限制（即时生效），用量追踪是可视化（管理员后台查看），不活跃跳过是自动节省（无需手动操作）。

## DD-008: 定时任务多用户遍历 + 随机延迟

**背景**：所有定时任务（晨报、日报、待办提醒等）需要为每个活跃用户执行。如果 3 个用户同时触发，会在极短时间内打出 3 个 LLM 请求，可能触发 API 限流。

**决策**：`/system` 端点收到定时任务请求后，遍历 `get_all_active_users()` 返回的活跃用户列表，为每个用户独立执行。用户之间加 `random.uniform(1, 3)` 秒随机延迟。单个用户执行失败（try/except）不影响其他用户。

```python
for uid in active_users:
    try:
        ctx = get_or_create_user(uid)
        do_action(uid, ctx)
    except Exception:
        logger.error(f"[{action}] {uid} 失败", exc_info=True)
    time.sleep(random.uniform(1, 3))
```

**理由**：随机延迟将 3 个并发请求分散到 3-9 秒内，避免 DeepSeek API 返回 429。try/except 隔离确保某用户的数据异常不会拖垮整个定时任务。

## DD-009: 新用户自动注册 + 三步引导

**背景**：需要让新用户发第一条消息就能开始使用，无需任何额外配置。

**决策**：`get_or_create_user(user_id)` 检测新用户后，自动完成：
1. 创建完整目录树（11 个目录）
2. 写入默认文件（Quick-Notes.md / Todo.md / state.json / memory.md / user_config.json）
3. 注册到 `users.json`

然后进入三步 Onboarding 引导：
- **Step 1**：发欢迎消息，引导设置昵称 → LLM 从回复中提取昵称
- **Step 2**：引导发送第一条笔记 → 展示速记能力
- **Step 3**：引导创建第一个待办 → 展示待办管理
- **Step 0**：引导完成，正常使用

Onboarding 进度存储在 `user_config.json` 的 `onboarding_step` 字段，支持中断续接。

**理由**：零配置上手是核心原则。自动注册 + 渐进引导让新用户在 3 条消息内了解核心功能。`_registry_lock`（threading.Lock）保证并发场景下不会为同一用户重复创建。

## DD-010: V8 / V6 / 节奏学习从 Karvis 移植策略

**背景**：Karvis 持续迭代新能力（V8 智能调度、V6 动态引擎、用户节奏学习），KarvisForAll 需要同步移植。两个项目的核心差异：

| 差异点 | Karvis | KarvisForAll |
|--------|--------|-------------|
| 用户模型 | 单用户，全局 state | 多用户，per-user state via ctx |
| 存储 | OneDrive | 本地磁盘 |
| Handler 签名 | `(params, state)` | `(params, state, ctx)` |
| 调度器 | 全局一份 state | 每用户独立 intents/rhythm |

**移植策略**：

1. **V8 智能调度引擎**（~300 行）：`_generate_daily_intents` / `_scheduler_tick` / `_rule_evaluate` / `_try_merge_intents` / `_execute_intent` 函数签名统一加 `(uid, ctx)` 参数。意图队列和节奏画像存储在各用户的 `state.scheduler` 字段中。`_daily_init` 和 `scheduler_tick` 遍历所有活跃用户逐一处理。
2. **V6 动态引擎**（~270 行）：`execute()` 签名加 `ctx=None`，文件操作通过 `ctx.base_dir` 解析用户目录。白名单安全控制不变。`from storage import IO` 引用本地 IO。
3. **用户节奏学习**（~60 行）：`_update_user_rhythm(state)` 不需要 ctx（操作的是 state 对象）。时间窗口过滤（wake: 05:00-12:00, sleep: 20:00-04:00）防止脏数据。滑动 7 天加权平均。

**理由**：增量移植，每个能力独立一次 PR。核心逻辑不变，只改入口签名和路径解析。移植后功能与 Karvis 保持一致，但天然支持多用户。

## DD-011: Git Push 云服务器部署

**背景**：KarvisForAll 部署在腾讯云 Lighthouse 服务器上（119.29.237.199），需要一种简单的代码更新方式。

| 方案 | 优点 | 缺点 |
|------|------|------|
| 手动 scp 上传 | 简单 | 每次手动操作，易遗漏文件 |
| CI/CD（GitHub Actions） | 全自动 | 需要 GitHub 仓库 + 配置 |
| git push + post-receive 钩子 | 一条命令部署，零外部依赖 | 需要服务器上配置 bare repo |

**决策**：方案三。服务器上创建 bare repo `~/karvis-repo.git`，配置 `post-receive` 钩子：

```bash
# 本地 .git/config
[remote "deploy"]
    url = root@119.29.237.199:~/karvis-repo.git
```

`post-receive` 钩子自动执行：
1. `git checkout -f` 拉取最新代码到工作目录
2. `pip install -r requirements.txt` 安装依赖
3. 停止旧进程 + 启动新进程
4. 输出部署完成 + PID

**部署命令**：`git push deploy master`

**理由**：一条命令完成部署，零外部依赖。与 Karvis 的 `git push prod main` 保持一致的运维体验。post-receive 钩子可定制化（重启方式、日志路径等）。

## DD-012: Skill 自动发现（热加载）

**背景**：KarvisForAll 继承自 Karvis 的 Skill 热加载机制（DD-011 in Karvis）。每新增一个 Skill 无需修改任何已有文件。

**决策**：每个 skill 模块末尾声明 `SKILL_REGISTRY = {"skill.name": handler_fn}` 字典。`skill_loader.py` 使用 `importlib` 扫描 `skills/` 目录，自动发现并合并所有注册表。

```python
# skill_loader.py 核心逻辑
for filename in sorted(os.listdir(skills_dir)):
    if filename.endswith(".py") and not filename.startswith("_"):
        module = importlib.import_module(f"skills.{filename[:-3]}")
        if hasattr(module, "SKILL_REGISTRY"):
            registry.update(module.SKILL_REGISTRY)
```

首次加载后缓存到 `_cached_registry`。`sorted()` 保证加载顺序确定性。重复 skill_name 会覆盖并打印警告。

**理由**：新增 skill 只需创建一个 `.py` 文件并在末尾写一行 `SKILL_REGISTRY`，零改动已有代码。当前已注册 37 个 skill handler。

## DD-013: 健康检查日志过滤

**背景**：Nginx 和 Docker 的健康检查每隔几秒请求 `GET /` 或 `GET /health`，导致 Werkzeug 的 access log 被大量无用日志刷屏，淹没真正的业务日志。

**决策**：在 Werkzeug logger 上添加自定义 `_HealthCheckFilter`：

```python
class _HealthCheckFilter(logging.Filter):
    def filter(self, record):
        msg = record.getMessage()
        return '"GET / ' not in msg and '"GET /health' not in msg
```

在 `app.py` 启动时注册：`logging.getLogger("werkzeug").addFilter(_HealthCheckFilter())`

**理由**：只过滤特定路径的 access log，不影响其他日志。过滤逻辑简单高效（字符串包含检查），零性能开销。

## DD-014: 先回复后保存（Reply-Before-Save）

**背景**：继承自 Karvis（DD-010）。`brain.process()` 在 LLM 决策完成后，需要写入 state/memory/decision_log。如果同步写完再回复，用户多等 1-3 秒（本地写入比 OneDrive 快很多，但仍有开销）。

**决策**：`brain.process()` 接受可选的 `send_fn` 回调。LLM 决策完成后，立即通过 `send_fn(reply)` 发送回复，然后再执行 state/memory/决策日志的并发写入。返回 `{"reply": ..., "already_sent": bool}` 告知调用方是否已发送，防止双发。

**理由**：与 Karvis 保持一致的优化策略。虽然本地写入比 OneDrive 快得多，但先发后写仍能节省用户 100-500ms 的等待时间。

## DD-015: 三层多模型路由

**背景**：继承自 Karvis（DD-017），针对不同复杂度任务使用不同模型。

**决策**：三层模型架构 + 静态路由。

| 层级 | 模型 | 用途 |
|------|------|------|
| Flash | Qwen（阿里云百炼） | companion_check、Flash 智能回复 |
| Main | DeepSeek V3.2 thinking=off | 用户消息路由、系统报告 |
| Think | DeepSeek V3.2 thinking=on | deep_dive、decision_track |

**统一入口**：`brain.call_llm(messages, model_tier="main"|"flash"|"think")`。Flash 失败自动降级到 DeepSeek Main。

**路由策略**：静态映射。`_select_model_tier(payload)` 根据 action 类型决定；`_select_skill_model_tier(skill_name)` 根据 skill 名决定。

**KarvisForAll 差异**：Qwen API Key 为可选配置。未配置时全部走 DeepSeek，Flash 层自动降级。

## DD-016: 对话压缩

**背景**：继承自 Karvis（DD-013）。短期记忆 FIFO 窗口为 10 条，溢出时需要压缩保留上下文。

**决策**：当 `recent_messages` 超过 10 条时，将最旧消息压缩为一条 `role=system` 的 `[对话摘要]`（纯文本拼接，包含时间范围和每条消息的摘要），保留最近 4 条原始消息。不额外调用 LLM。

**理由**：零额外 LLM 调用，零延迟开销。摘要虽不如 LLM 精细，但保留了时间线和核心内容。多用户场景下避免额外的 LLM 成本尤为重要。

## DD-017: V8 智能调度引擎 — 意图队列 + 规则引擎

**背景**：继承自 Karvis（DD-025），但需要多用户适配。原有固定 cron 不考虑各用户的实际作息差异。

**决策**：意图队列 + 心跳驱动 + 8 条规则引擎 + 意图合并 + 用户节奏自学习。

**核心架构**：
1. **意图队列**：每天 05:00 为每个活跃用户生成 6 个触达意图（morning_report / todo_remind / companion / nudge_check / evening_checkin / daily_report）
2. **心跳驱动**：每 30 分钟 `scheduler_tick` 遍历所有活跃用户，检查到期意图
3. **规则引擎**：时间窗口 → 用户节奏 → 安静时段 → 刚聊完不推 → 沉默时长条件 → 次数限制 → ideal 时间 → 兜底 latest
4. **意图合并**：`(evening_checkin, daily_report)` 和 `(morning_report, todo_remind)` 可合并
5. **节奏自学习**：每条消息后 `_update_user_rhythm(state)` 推算起床/入睡时间

**多用户适配**：
- `_daily_init(uid, ctx)` 为每个用户独立初始化意图队列
- `_scheduler_tick(uid, ctx)` 逐用户评估，用户间加随机延迟
- 意图队列和节奏画像存储在各用户的 `state.scheduler` 中

**APScheduler 任务**：
| 任务 | 触发方式 |
|------|----------|
| `daily_init` | cron 05:00 |
| `scheduler_tick` | interval 30min |
| `refresh_cache` | interval 30min |
| `mood_generate` | cron 22:00 |
| `weekly_review` | cron 周日 21:30 |
| `monthly_review` | cron 月末 22:00 |

**理由**：每个用户有独立的作息节奏，一套固定 cron 无法满足。意图队列让每个用户获得个性化的推送时间。规则引擎 0ms 开销，不需要额外 LLM 调用。

## DD-018: Storage 策略模式

**背景**：Karvis 使用 OneDrive Graph API（`OneDriveIO`），KarvisForAll 使用本地磁盘（`LocalFileIO`）。大量历史代码通过 `from storage import IO as OneDriveIO` 引用存储层。

**决策**：`storage.py` 作为策略模式入口，一行代码切换后端：

```python
from local_io import LocalFileIO
IO = LocalFileIO
STORAGE_MODE = "local"
```

所有模块统一引用 `from storage import IO`。保留 `IO` 别名和兼容接口（如 `get_token()` 在本地模式返回 `"local"`），历史代码中的 `OneDriveIO` 别名无需改动。

**`LocalFileIO` 关键设计**：
- `threading.Lock()` 保护写操作
- `_resolve_path()` 直接使用 ctx 提供的绝对路径
